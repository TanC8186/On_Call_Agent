## 一、架构设计

### Q1.1 这个项目的整体架构是怎样的？画出组件拓扑图并说明各组件职责。

**考察点**: 系统设计能力，对分布式系统的理解。

**参考要点**:
- FastAPI Web Server（端口 9900）：路由、SSE 流式、静态文件
- LangGraph 工作流引擎：Plan-Execute-Replan 三节点
- MCP Servers（端口 8003/8004）：CLS 日志服务 + Monitor 监控服务，基于 fastmcp
- Milvus 向量数据库（Docker，端口 19530）：RAG 知识库存储
- DeepSeek API：LLM 推理（通过 OpenAI 兼容协议）
- 本地 Embedding 模型：sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2

### Q1.2 为什么选择 Plan-Execute-Replan 而不是传统的 ReAct 模式？

**考察点**: Agent 架构选型能力。

**参考要点**:
- ReAct 适用于简单、短链路的单轮工具调用
- Plan-Execute-Replan 适用于需要多步骤、有依赖关系的复杂诊断任务
- Plan 阶段可以检索知识库获取经验，制定结构化计划
- Replan 阶段可以评估执行结果，决定继续/调整/终止
- 有明确的步骤上限（MAX_STEPS=8）防死循环

### Q1.3 项目中有两个 Agent（Chat Agent 和 AIOps Agent），为什么不做成统一的 Agent？

**考察点**: 架构权衡能力。

**参考要点**:
- Chat Agent 用 LangGraph 内置 `create_agent`（ReAct 模式），适合自由对话
- AIOps Agent 用自定义 StateGraph（Plan-Execute-Replan），适合结构化诊断
- 两者有不同的状态定义、System Prompt、温度参数、上下文策略
- 分离可降低耦合，各自独立优化

### Q1.4 三个 MCP Server 之间是什么关系？为什么用 MCP 协议而不是直接 import 调用？

**考察点**: 协议选型、服务解耦理解。

**参考要点**:
- MCP 解耦了工具提供方和消费方，支持独立部署和语言无关
- `langchain-mcp-adapters` 的 `MultiServerMCPClient` 统一管理多服务器
- 支持 streamable-http 传输，可跨机器部署
- 自带工具发现机制（`get_tools()`），新增工具无需改 Agent 代码

## 二、LLM 与模型

### Q2.1 为什么用 DeepSeek 而不是直接用 OpenAI？DeepSeek 有什么限制需要特别处理？

**考察点**: 模型选型能力、对 API 差异的理解。

**参考要点**:
- DeepSeek 性价比高，中文能力强
- **关键限制**：不支持 `response_format`（JSON Mode / structured output）
- 解决方案：用 plain LLM + 手动 JSON 解析（`_parse_json_from_text()`）
- 解析策略：直接解析 → Markdown code block 提取 → 正则花括号匹配 → 逐行回退
- 使用 `.replace()` 而非 `.format()` 避免工具描述中的 `{` 冲突

### Q2.2 `_parse_json_from_text()` 的设计思路是什么？有哪些容错策略？

**考察点**: 鲁棒性设计、正则表达式能力。

**参考要点**（4 层回退）:
```python
1. json.loads(text.strip())                          # 直接解析
2. re.search(r'```(?:json)?\s*([\s\S]*?)\s*```')    # Markdown 代码块
3. re.search(r'\{[\s\S]*\}')                         # 正则花括号匹配
4. 逐行文本解析，抽取有效步骤                           # 最终回退
```

### Q2.3 Planner 中为什么用 `.replace()` 而不是 `.format()` 或 f-string？

**考察点**: 细节敏感度。

**参考要点**:
- 工具描述（tools_description）中可能包含 `{` 和 `}` 字符
- 如果用 `.format()`，这些字符会被解释为占位符导致异常
- `.replace("{key}", value)` 只替换明确的占位符，更安全

### Q2.4 Chat Agent 用 temperature=0.7，AIOps 用 temperature=0，为什么不同？

**考察点**: 对 temperature 参数的理解。

**参考要点**:
- Chat Agent 面向开放式对话，需要一定创造性，0.7 提供多样性
- AIOps 需要确定性输出，0 最大化稳定性，减少幻觉
- Planner/Replanner 输出 JSON，0 保证格式一致性

## 三、Multi-Agent 工作流

### Q3.1 Plan-Execute-Replan 三个节点各做什么？它们之间如何传递状态？

**考察点**: LangGraph StateGraph 理解。

**参考要点**:
- **Planner**: 检索知识库 → 获取工具列表 → LLM 生成 JSON 计划 → 输出 `{plan: [step1, step2...]}`
- **Executor**: 取 plan[0] → LLM 绑定工具 → ToolNode 执行 → 输出 `{plan: plan[1:], past_steps: [(task, result)]}`
- **Replanner**: 评估执行结果 → 决策 continue/replan/respond → 输出 `{response: "..."}` 或 `{plan: [...]}`
- 状态通过 `PlanExecuteState(TypedDict)` 在节点间传递
- `past_steps` 使用 `operator.add` 实现追加而非覆盖
- `response` 非空作为终止信号

### Q3.2 这个工作流有哪些防死循环机制？

**考察点**: 鲁棒性设计。

**参考要点**:
- `MAX_STEPS = 8`：全局硬限制，超限强制 respond
- `past_steps >= 5`：禁止 replan，只能 continue 或 respond
- Replan 时新步骤数必须 ≤ 剩余步骤数，否则截断
- Planner 输入截断 500 字符，防止超长 prompt

### Q3.3 Replanner 中 "respond" / "continue" / "replan" 三种决策是如何在 Prompt 中引导的？

**考察点**: Prompt Engineering 能力。

**参考要点**:
- 按优先级排序：respond > continue > replan
- 明确决策标准：已执行 ≥ 3 步且获取关键信息 → respond
- "不要等到'完美'才响应，'足够好'就应该立即 respond"
- replan 被严格限制："新步骤数量必须 ≤ 当前剩余步骤数"
- 强制 JSON 输出，避免自由格式的不确定性

### Q3.4 `should_continue()` 条件边是如何工作的？什么情况下工作流会终止？

**考察点**: LangGraph 条件路由理解。

**参考要点**:
```python
def should_continue(state):
    if state.get("response"):        # 最终响应已生成
        return END
    if state.get("plan"):            # 还有剩余步骤
        return "executor"            # 继续循环
    return END                       # 计划空且无响应（异常情况）
```

### Q3.5 如果要加一个"人工确认"节点，让用户在关键决策点介入，应该在什么位置插入？

**考察点**: 工作流设计扩展能力。

**参考要点**:
- 插入位置：Replanner 之后、Executor 之前
- 当 Replanner 决策为高风险操作时（如重启服务），暂停并等待前端确认
- 用 LangGraph 的 `interrupt` 机制或 SSE 双向通信实现
- 需要在 State 中增加 `pending_approval` 字段

## 四、RAG 知识库

### Q4.1 文档上传到向量检索的完整链路是什么？

**考察点**: 数据管线理解。

**参考要点**:
```
POST /api/upload
  → 文件验证（扩展名 .txt/.md，大小 ≤ 10MB）
  → 保存到 ./uploads/
  → vector_index_service.index_single_file()
  → 读取 UTF-8 内容
  → 删除旧 Milvus 数据（按 _source 过滤）
  → document_splitter_service.split_document()
  → 三阶段切分
  → vector_store_manager.add_documents()
  → vector_embedding_service.embed_documents()
  → Milvus insert
```

### Q4.2 文档切分的三阶段流水线是什么？每个阶段解决什么问题？

**考察点**: 文档处理设计能力。

**参考要点**:
```
阶段1: MarkdownHeaderTextSplitter(headers=["#", "##"])
       → 语义切分，保留文档结构（h1/h2 作为元数据）

阶段2: RecursiveCharacterTextSplitter(chunk_size=1600, overlap=100)
       → 大小切分，控制单块长度，overlap 保持上下文连续

阶段3: _merge_small_chunks(min_size=300)
       → 合并碎片，防止过度切分导致语义不完整
```

### Q4.3 为什么阶段2用 chunk_size=1600 而不是配置的 800？

**考察点**: 二次切分的逻辑理解。

**参考要点**:
- 阶段2 是 `chunk_size * 2 = 1600`
- 目的：减少分片数，避免过度碎片化
- 阶段1 已经做了语义切分，阶段2 只处理超长块
- 这是一个调优参数：如果 chunk 数量太多导致检索慢，可以增大这个值

### Q4.4 向量维度（384）是如何确定的？如果换了 Embedding 模型怎么办？

**考察点**: 向量数据库运维能力。

**参考要点**:
- 由 Embedding 模型输出维度决定：paraphrase-multilingual-MiniLM-L12-v2 → 384 维
- `milvus_client.py` 在 `connect()` 时自动检测维度不匹配
- 如果现有 collection 维度 ≠ VECTOR_DIM，自动 drop + recreate
- 切换模型的步骤：1) 改 `.env` 中 EMBEDDING_MODEL → 2) 改 VECTOR_DIM → 3) 重启，旧 collection 自动重建
- 代价：旧数据丢失，需重新索引

### Q4.5 `retrieve_knowledge` 工具的召回策略是什么？Top-K 如何确定？

**考察点**: RAG 检索策略。

**参考要点**:
- 向量相似度：L2 欧氏距离
- 索引类型：IVF_FLAT, nlist=128
- Top-K：默认 3（`.env` 中 `RAG_TOP_K=3`）
- 无 Reranker：直接按 L2 距离排序
- `response_format="content_and_artifact"`：同时返回格式化文本和原始 Document

### Q4.6 如果 Top-3 召回结果都不相关怎么办？如何优化？

**考察点**: RAG 优化思路。

**参考要点**:
- 增大 Top-K 到 5 或 10
- 加 Reranker（如 bge-reranker-v2-m3）对 Top-10 重排序
- 混合检索：向量 + BM25 关键词检索
- 优化 Embedding 模型：换更强的模型（如 bge-large-zh-v1.5）
- 优化 chunk 策略：调整 chunk_size 和 overlap
- 加入 Query Rewriting：用 LLM 改写用户问题后再检索
- Milvus 索引升级：IVF_FLAT → IVF_SQ8 或 HNSW

## 五、MCP 协议

### Q5.1 MCP 在这个项目中扮演什么角色？为什么用 fastmcp？

**考察点**: 协议理解、框架选型。

**参考要点**:
- MCP（Model Context Protocol）：LLM 应用与外部工具/数据源的标准协议
- 本项目中 CLS Server 和 Monitor Server 都是 MCP Server
- Agent 通过 `MultiServerMCPClient` 统一发现和调用工具
- fastmcp：Python MCP 框架，简化 Server 开发（装饰器即工具）

### Q5.2 MCP 工具调用失败怎么办？重试策略是什么？

**考察点**: 容错设计。

**参考要点**:
- `retry_interceptor` 拦截所有 MCP 工具调用
- 指数退避：1s → 2s → 4s，最多 3 次
- 全部失败后返回 `CallToolResult(isError=True)` 而非抛异常
- Agent 可以感知到错误，在最终报告中如实说明

### Q5.3 如何新增一个 MCP 工具？需要改哪些文件？

**考察点**: 扩展能力。

**参考要点**:
1. 在对应 MCP Server 中添加 `@mcp.tool()` 装饰的函数
2. 重启 MCP Server
3. **无需改 Agent 代码**：Agent 通过 `get_tools()` 自动发现新工具
4. 工具描述会通过 `format_tools_description()` 注入 Planner Prompt

### Q5.4 Monitor MCP Server 中真实 API 和模拟数据是如何切换的？

**考察点**: 降级策略。

**参考要点**:
```python
USE_REAL_API = bool(SECRET_ID and SECRET_KEY)
if not USE_REAL_API:
    return _mock_metric_data(...)  # 返回模拟数据
# 否则调用腾讯云 Monitor API
```
- 有凭证 → 真实腾讯云 API
- 无凭证 → 模拟数据（带 `source: "mock"` 标记）
- 这个设计支持不依赖云环境的本地开发调试

## 六、工程化与运维

### Q6.1 项目是如何做错误处理的？有哪些关键的保护层？

**考察点**: 工程健壮性。

**参考要点**（8 层保护）:
| 层级 | 策略 |
|------|------|
| Planner 异常 | 回退到默认计划 |
| Executor 异常 | 记录失败但继续下一步 |
| Replanner 异常 | 返回空 dict，LangGraph 自动继续 |
| MCP 不可用 | 静默跳过，仅用本地工具 |
| Milvus 不可用 | 警告但不阻塞，RAG 降级 |
| 工具调用失败 | 3 次指数退避重试 |
| SSE 流异常 | yield error 事件，不断开连接 |
| AIOps 全局异常 | yield error 事件，终止工作流 |

### Q6.2 Embedding 模型加载失败怎么处理？`local_files_only` 为什么优先？

**考察点**: 国内网络环境下的工程经验。

**参考要点**:
- hf-mirror.com 不稳定，下载可能超时
- 优先 `local_files_only=True`：直接使用本地缓存
- 失败后回退到在线模式（会尝试 hf-mirror.com）
- `HF_ENDPOINT` 必须在任何 `huggingface_hub` 导入前设置
- 所以在 `vector_embedding_service.py` 和 `web_server.py` 的最顶部设置

### Q6.3 Windows 环境下遇到过哪些坑？如何解决的？

**考察点**: 跨平台经验。

**参考要点**:
- 控制台 GBK 编码：emoji/UTF-8 字符导致崩溃 → 禁用 loguru 彩色输出
- Batch 文件编码：UTF-8 BOM 导致 `'锛?' 不是内部或外部命令` → 手动启动
- 路径分隔符：Unix `/` vs Windows `\` → 用 `pathlib.Path` 统一处理
- Docker：国内镜像拉取需用 `docker.1ms.run` 中转

### Q6.4 为什么要设置 `os.environ["HF_ENDPOINT"]` 在所有 import 之前？

**考察点**: Python import 时序理解。

**参考要点**:
- `huggingface_hub` 库在 import 时就会读取 `HF_ENDPOINT` 环境变量并缓存
- 如果在 import 之后才设置，库已经用了默认的 `https://huggingface.co`
- 所以在 `web_server.py` 第一行就设置，在 `vector_embedding_service.py` 模块顶部设置
- 这是 Python 环境变量注入的经典时序问题

## 七、幻觉与安全

### Q7.1 项目中通过哪些手段防止 LLM 幻觉？

**考察点**: AI 安全实践。

**参考要点**:
1. **System Prompt 约束**："不要编造数据，只返回实际获取的信息"
2. **Temperature=0**：AIOps 三节点都用 0，最大化确定性
3. **强制引用证据**：`{execution_history}` 传入，必须基于工具实际返回值
4. **步骤上限**：MAX_STEPS=8 防止无限循环中不断编造
5. **JSON 格式约束**：结构化输出减少自由发挥空间

### Q7.2 如果用户输入"帮我删掉数据库"或者注入 Prompt，如何防御？

**考察点**: 安全设计。

**参考要点**:
- 当前防御：工具均为**只读**（retrieve_knowledge, get_current_time, MCP 查询类工具）
- Agent 没有写入/删除能力
- MCP Server 的工具函数都是查询操作
- **不足**：没有输入过滤层、没有 Prompt Injection 检测
- **建议**：增加输入 sanitize、敏感操作需二次确认

### Q7.3 `.env` 中包含 API Key，生产环境应该怎么管理？

**考察点**: 安全最佳实践。

**参考要点**:
- 不应提交到 Git（加入 `.gitignore`）
- 生产环境用：K8s Secrets / 云密钥管理服务 / CI/CD 注入环境变量
- `.env` 仅用于本地开发
- 建议增加 `.env.example` 模板文件（不含真实密钥）

## 八、Prompt Engineering

### Q8.1 Planner 的 Prompt 经历了哪些调优？为什么加了经验文档占位符？

**考察点**: Prompt 迭代经验。

**参考要点**:
- 初始版本：只有 `{tools_description}`，无经验参考
- 优化后：增加了 `{experience_context}` 占位符
- 效果：Planner 能从知识库中检索相似问题的解决方案，制定更精准的计划
- 还加了 JSON 示例："示例输出: {"steps": [...]}"，指导输出格式

### Q8.2 Replanner 的 Prompt 为什么要强调"不要等到完美才响应"？

**考察点**: Agent 行为调优经验。

**参考要点**:
- 实际使用中发现 Replanner 倾向过度分析，不断 replan 而不给出最终答案
- "足够好就应该立即 respond" 是抑制完美主义的经验性约束
- 配合步骤数阈值（≥3 步 + 关键信息获取 → respond）形成双重约束

### Q8.3 为什么 Prompt 中使用 `.replace()` 而不是 `.format()`？

**考察点**: 细节经验。

**参考要点**:
- 工具描述可能包含 `{` 和 `}`（如 JSON schema 示例）
- `.format()` 会将这些字符误认为占位符，抛出 KeyError
- `.replace()` 是精确字符串替换，更安全

## 九、性能与优化

### Q9.1 向量检索慢怎么排查和优化？

**考察点**: 性能调优能力。

**参考要点**:
1. **排查**：检查 chunk 数量、查询耗时分布、Milvus 索引状态
2. **索引优化**：IVF_FLAT → IVF_SQ8（量化压缩）或 HNSW（图索引）
3. **分片扩容**：`num_shards` 从 2 增加到 4
4. **Embedding 缓存**：对高频 query 做 LRU 缓存
5. **混合检索**：向量 + BM25 关键词，减少纯向量检索范围
6. **Reranker**：先粗召回（Top-20），再精排（Top-3）

### Q9.2 SSE 流式输出中如何做背压控制？

**考察点**: 流式系统经验。

**参考要点**:
- LangGraph 的 `astream(stream_mode="messages")` 内置流控
- Chat Agent 的 `trim_messages_middleware` 限制上下文窗口到 7 条
- ToolNode 执行时自动等待工具返回后再继续流式输出
- 如果 LLM 生成速度 > 网络发送速度，SSE 天然提供背压

### Q9.3 内存管理：会话历史存在哪里？服务重启会丢失吗？

**考察点**: 状态管理。

**参考要点**:
- Chat Agent 和 AIOps Agent 各有一个 `MemorySaver()` checkpointer
- **存储在内存**，服务重启丢失
- 以 `thread_id`（前端传入的 session_id）隔离会话
- 前端可调用 `/api/chat/clear` 主动清空
- **改进方向**：用 `SqliteSaver` 或 `PostgresSaver` 持久化

## 十、评价与迭代

### Q10.1 如果没有人工反馈，你打算用什么指标评价这个 Agent 的好坏？

**考察点**: 评价体系设计。

**参考要点**:
1. **任务完成率**：最终响应是否覆盖所有子问题（LLM-as-Judge）
2. **工具选择准确率**：Planner 指定的工具 vs Executor 实际调用的工具匹配度
3. **响应时延**：P50/P95/P99 端到端时延
4. **幻觉率**：响应中工具未返回的"事实"占比
5. **步骤效率**：同类问题的平均执行步数
6. **检索命中率**：Recall@K
7. **一致性**：相同问题在不同轮次的输出一致性

### Q10.2 如何进行 Prompt 的 A/B 测试？

**考察点**: 实验设计。

**参考要点**:
- 准备固定的测试问题集（10-20 个覆盖各种场景的问题）
- 变体 A（当前 Prompt）vs 变体 B（新 Prompt）
- 每个变体跑 3 次取平均值（减少 LLM 随机性影响）
- 比较：步骤数、最终评分（LLM-as-Judge）、工具调用准确率
- 统计显著性检验（如果样本够大）

### Q10.3 如果要做一个 Demo 场景模式，模拟运维告警来演示全链路，你会怎么设计？

**考察点**: 演示/测试设计能力。

## 十一、场景题

### Q11.1 实际运维中，AIOps Agent 执行了 5 步还没找到根因，此时应该怎么办？

**参考要点**:
- Replanner 在 `past_steps >= 5` 时会强制 respond（代码已实现）
- 报告应诚实说明：当前证据、已排除的原因、建议人工介入的方向
- 不应继续编造或循环

### Q11.2 如果用户问了一个和运维完全无关的问题（如"推荐一首歌"），Agent 怎么处理？

**参考要点**:
- Chat Agent 的 System Prompt 定义为"专业 AI 助手"，会按通用能力回答
- AIOps Agent 会被固定诊断模板约束，不受用户自由输入影响
- 当前没有硬性拒绝机制，依赖 LLM 自身判断
- 改进：增加意图识别层，非运维问题返回礼貌拒绝

### Q11.3 假设 Embedding 模型升级了（e.g. 384维 → 1024维），如何平滑迁移？

**参考要点**:
1. 修改 `VECTOR_DIM` 和 `EMBEDDING_MODEL` 配置
2. 重启服务 → `milvus_client.connect()` 自动检测维度不匹配
3. 自动 drop 旧 collection + create 新 collection
4. **重新索引所有文档**（`POST /api/index_directory`）
5. 迁移窗口期 RAG 功能不可用，但对话功能不受影响（代码中 vector_store 是懒加载 + 降级）

### Q11.4 用户上传了一个 50MB 的 Markdown 文件，系统会怎么处理？

**参考要点**:
- `MAX_FILE_SIZE = 10MB`，50MB 会被直接拒绝（`file.py:63-64`）
- 返回 HTTP 400："文件大小超过限制"
- 即使通过了大小检查，也只支持 `.txt` 和 `.md` 扩展名
- 这是多层防御：前端限制 + 后端验证

### Q11.5 如果两个用户同时上传文件并查询，会不会互相干扰？

**参考要点**:
- 文档上传：写入同一 collection，通过 `_source` metadata 区分文件
- 会话隔离：Chat Agent 通过 `thread_id` 隔离，AIOps Agent 通过 `session_id` 隔离
- 检索：`retrieve_knowledge` 返回的是全局检索结果，不按用户过滤
- 潜在问题：用户 A 上传的敏感文档可能被用户 B 检索到
- 改进：增加用户维度的 Milvus Partition Key 隔离
